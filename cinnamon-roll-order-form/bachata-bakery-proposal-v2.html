System Overview

Bachata Bakery's Cinnamon Roll Order Form is a time-gated, cap-enforced pre-order system with Square card payment integration. Customers order online during a defined weekly window, pay immediately via credit or debit card, and the bakery closes the form automatically once capacity is reached — with zero manual intervention needed.

Rule 1

🗓 Time Gate

Orders accepted Friday 12:00 AM – Sunday 11:59 PM only. Outside this window the form shows a live countdown to next opening.

Rule 2

📦 Dual Cap

Form closes when 100 rolls are sold OR 50 customer orders are placed — whichever comes first.

Rule 3

💳 Square Only

All payments processed via Square Payments API. Cards charged immediately on submission. No partial holds, no manual verification.

✅ Simplification from v1: By removing Zelle, every order is either paid or declined — no pending states, no cron cleanup jobs, no admin verification workflow. This cuts complexity significantly and gives the bakery clean, real-time fulfillment data.

Core Logic

Time Gate (Server-Side Enforcement)

The backend is always authoritative. Even if JavaScript is bypassed, a direct POST to the API will be rejected outside the window.

// api/helpers/time_gate.php — uses bakery's configured timezone
function isFormOpen(): bool {
    $tz  = new DateTimeZone('America/New_York');
    $now = new DateTime('now', $tz);
    $dow = (int) $now->format('N');  // 5=Fri, 6=Sat, 7=Sun
    $h   = (int) $now->format('G');
    $m   = (int) $now->format('i');

    if ($dow === 5 || $dow === 6) return true;
    if ($dow === 7  && ($h < 23 || ($h === 23 && $m <= 59))) return true;
    return false;
}

Atomic Cap Check + Square Charge

The cap is checked and the order is inserted in a single database transaction with a row lock, preventing two customers from simultaneously purchasing the last slot.

// api/order.php — all-or-nothing atomic flow
$pdo->beginTransaction();

$cap = $pdo->prepare("SELECT rolls_sold, orders_placed FROM order_caps WHERE window_id = :wid FOR UPDATE");
$cap->execute([':wid' => $windowId]);
$row = $cap->fetch();

if ($row['orders_placed'] >= 50)             jsonError('sold_out_orders');
if ($row['rolls_sold'] + $qty > 100)          jsonError('sold_out_rolls');

// Cap check passed — charge the card via Square
$charge = squareCharge($nonce, $amountCents, $idempotencyKey);

if ($charge['status'] !== 'COMPLETED') {
    $pdo->rollBack();
    jsonError('card_declined', $charge['errors'][0]['detail']);
}

// Both cap and payment cleared — commit everything
$pdo->prepare("UPDATE order_caps SET rolls_sold = rolls_sold + :q, orders_placed = orders_placed + 1 WHERE window_id = :wid")->execute([...]);
$pdo->prepare("INSERT INTO orders (...) VALUES (...)")->execute([...]);
$pdo->commit();

sendConfirmationEmail($customer, $orderRef);

Contingency Planning

Scenario

Card Declined by Square

Transaction rolls back atomically. No order record is created. Cap is not incremented. Customer sees a plain-English decline message with suggestion to retry a different card.

Scenario

PHP Times Out After Charge

Square's webhook fires payment.completed. Webhook handler reconciles the orphaned charge by finding the Square payment ID and creating the missing order record. Owner gets an email alert.

Scenario

Two Customers Hit Cap Simultaneously

FOR UPDATE row lock serializes the requests. Second request blocks, reads the updated cap, and returns a sold-out error. Guaranteed no oversell.

Scenario

Double-Click Submit

The submit button is disabled immediately on first click. The idempotency key (unique per session) also prevents Square from charging twice even if two identical requests reach the API.

Scenario

Owner Wants to Close Early

Admin dashboard has a Force Close toggle. Sets a DB flag read by status.php. Form closes within 60 seconds on all active customer sessions. Fully reversible.

Scenario

Owner Wants to Raise/Lower Cap

The order_caps table stores rolls_max and orders_max per weekend. Admin dashboard exposes editable fields — no code changes or deployments needed.
