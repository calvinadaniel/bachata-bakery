Migration Overview

This guide covers the complete path from a finished local Claude Code build to a verified, live production deployment on Hostinger. It is split into five stages — each must be completed and confirmed before moving to the next.

Stage 1–2

Pre-Flight

Hostinger environment setup, MySQL database creation, SMTP email account, and SSL verification — all before a single file is uploaded.

Stage 3

File Transfer

Transfer project files via SFTP, configure .env with live credentials, set correct file permissions, and lock down sensitive directories via .htaccess.

Stage 4

Square Go-Live

Swap from Square Sandbox to Production credentials, register the webhook endpoint in Square Dashboard, run the $0.01 live charge verification test.

Stage 5

Final QA + Sign-Off

Run the full 17-case QA checklist against live production. Confirm emails, admin dashboard, cap enforcement, and form behavior. Sign off before announcing to customers.

⚠️ Never go live on a Friday. The form opens Friday midnight. Deploy and verify on Wednesday or Thursday so you have time to fix any production issues before the window opens.

Stage 1 — Hostinger Pre-Flight Setup

Complete all of these in Hostinger's hPanel before uploading any files.

1
Confirm PHP version is 8.x
hPanel → Advanced → PHP Configuration → set to PHP 8.1 or 8.2. The project uses named arguments and enum types that require 8.x minimum.
2
Create the MySQL database
hPanel → Databases → MySQL Databases → Create New Database. Note the database name, username, and password exactly — these go into .env.
# Values you'll need for .env
DB_HOST=localhost
DB_NAME=u123456_bachata_bakery    # Hostinger prefixes with account username
DB_USER=u123456_bbuser
DB_PASS=YourStrongPasswordHere
ℹ️ Tip: On Hostinger shared hosting, DB_HOST is always localhost. On VPS it may differ — check hPanel → Databases for the host string.
3
Run schema.sql to create the tables
hPanel → phpMyAdmin → select your database → SQL tab → paste and run schema.sql. Confirm both orders and order_caps tables appear in the left panel.
-- Confirm tables exist after import
SHOW TABLES;
-- Expected output:
-- order_caps
-- orders
4
Create a dedicated email account for orders
hPanel → Emails → Email Accounts → Create. Use something like orders@bachatabakery.com. This becomes your SMTP_USER in .env. PHPMailer will send through Hostinger's SMTP using port 465 with SSL.
SMTP_HOST=smtp.hostinger.com
SMTP_USER=orders@bachatabakery.com
SMTP_PASS=YourEmailPasswordHere
SMTP_PORT=465
!
Verify SSL certificate is active
hPanel → SSL → confirm the certificate is installed and active on your domain. Square's Web Payments SDK will refuse to load on HTTP — SSL is non-negotiable. If it shows "Expired" or "Not installed", renew it before proceeding.
🚫 Hard blocker: Square SDK will not initialize on a non-HTTPS origin. No SSL = no payments. Verify this first.
03

Stage 2 — File Transfer via SFTP

1
Connect to Hostinger via SFTP
Use FileZilla or your preferred SFTP client. Credentials are in hPanel → Advanced → SSH Access (or FTP Accounts for non-SSH shared plans).
# FileZilla connection settings
Host:     sftp://yourdomain.com
Username: your_hostinger_username
Password: your_hostinger_password
Port:     22   # SFTP (not FTP port 21)
2
Upload project files to public_html
Upload the entire bachata-bakery-orders/ directory into public_html/. Your file tree on Hostinger should look like:
public_html/
├── index.html
├── .htaccess
├── assets/
│   ├── css/styles.css
│   └── js/form.js
├── api/
│   ├── helpers/
│   ├── status.php
│   ├── order.php
│   └── webhook/
│       └── square.php
├── admin/
└── emails/
⚠️ Do NOT upload: .env (create it on-server), .env.example is fine, node_modules/ if any, CLAUDE.md files, any .git/ directory.
!
Create .env directly on the server
Never upload .env from your local machine — it contains secrets. Create it fresh on the server using Hostinger's File Manager or SSH.
# Via SSH on Hostinger
nano public_html/.env

# Paste your production .env contents:
APP_ENV=production
SQUARE_APP_ID=sq0idp-...              # Production App ID
SQUARE_LOCATION_ID=...
SQUARE_ACCESS_TOKEN=EAAAl...          # Production Access Token
SQUARE_WEBHOOK_SIG_KEY=...
DB_HOST=localhost
DB_NAME=u123456_bachata_bakery
DB_USER=u123456_bbuser
DB_PASS=YourStrongPasswordHere
SMTP_HOST=smtp.hostinger.com
SMTP_USER=orders@bachatabakery.com
SMTP_PASS=YourEmailPasswordHere
SMTP_PORT=465
OWNER_EMAIL=owner@bachatabakery.com
BAKERY_TIMEZONE=America/New_York
4
Set correct file permissions
Via SSH or FileZilla's "File Permissions" right-click menu:
# Standard permissions
chmod 644 public_html/*.html
chmod 644 public_html/.htaccess
chmod 644 public_html/assets/css/styles.css
chmod 644 public_html/assets/js/form.js
chmod 755 public_html/api/
chmod 644 public_html/api/*.php
chmod 755 public_html/admin/

# .env must be locked down — owner read only
chmod 600 public_html/.env
5
Verify .htaccess is blocking sensitive paths
In your browser, attempt to access these URLs directly. All should return 403 Forbidden:
https://yourdomain.com/.env                   # Must return 403
https://yourdomain.com/api/helpers/db.php     # Must return 403
https://yourdomain.com/api/helpers/           # Must return 403
🚫 If any of these return 200 or show file contents, stop and fix the .htaccess rules immediately before proceeding. Your database credentials are exposed.
04

Stage 3 — Square Production Switch

⚠️ During development you used Square Sandbox credentials. All sandbox charges are fake. You must swap to Production credentials before any real customer money can be collected. This stage does exactly that.
1
Get Square Production credentials
Log in at developer.squareup.com → your Application → click the Production tab (not Sandbox). Collect:
.env KeyWhere to Find It
SQUARE_APP_IDProduction tab → Application ID
SQUARE_LOCATION_IDProduction tab → Location ID (under your business location)
SQUARE_ACCESS_TOKENProduction tab → Production Access Token
SQUARE_WEBHOOK_SIG_KEYCreated in next step
2
Switch the SDK script tag from Sandbox to Production
In index.html (or wherever the SDK is loaded), update the script src:
<!-- REMOVE this sandbox line -->
<script src="https://sandbox.web.squarecdn.com/v1/square.js"></script>

<!-- REPLACE with production -->
<script src="https://web.squarecdn.com/v1/square.js"></script>
🚫 Easy mistake: If you deploy with the sandbox SDK URL, the card form will load but no real charges will process. Double-check this line.
3
Register the webhook endpoint in Square Dashboard
Square Dashboard → Developers → Webhooks → Add Endpoint:
Endpoint URL:  https://yourdomain.com/api/webhook/square.php
API Version:   2024-01-17  # Match the version used in your PHP cURL calls
Events to subscribe:
  ✅ payment.completed
  ✅ payment.failed
After saving, Square shows a Signature Key. Copy it into your server's .env as SQUARE_WEBHOOK_SIG_KEY.
!
Run the $0.01 live charge verification test
This is your final proof that real money moves correctly end-to-end. Use a real card you own.
  • Open the live form on production URL
  • Submit a real order for the minimum quantity — set amount_cents = 1 temporarily in your test, or price one item at $0.01
  • Enter your real card number in the Square card field
  • Submit — confirm the order is accepted and you see the success screen
  • Check your email — confirm the order confirmation email arrived
  • Check Square Dashboard → Payments — confirm a $0.01 charge appears
  • Check Hostinger phpMyAdmin → orders table — confirm a row exists with payment_status = 'paid'
  • Check Square Dashboard → Payments → refund the $0.01 charge immediately
  • Restore the real product pricing before announcing to customers
5
Verify webhook fires on the live charge
After the $0.01 test charge, go to Square Dashboard → Developers → Webhooks → your endpoint → Event Deliveries. You should see a payment.completed event with a 200 response from your server.
⚠️ If webhook returns non-200: Check Hostinger error logs (hPanel → Advanced → Error Logs) for the PHP error. Common cause: missing SQUARE_WEBHOOK_SIG_KEY in .env or a PHP syntax error in webhook/square.php.
05

Stage 4 — Hostinger Cron Setup

One cron job is needed: the auto-expiry script that closes the form at Sunday midnight and resets the weekend window. Set this up in hPanel before launch.

1
Add the cron job in hPanel
hPanel → Advanced → Cron Jobs → Create New Cron Job.
# Runs every hour — checks for Sunday midnight window close
Frequency: Every hour (0 * * * *)
Command:   /usr/bin/php /home/u123456/public_html/api/cron/window-reset.php

# Alternatively, run once daily at 12:05 AM
Frequency: 5 0 * * *
Command:   /usr/bin/php /home/u123456/public_html/api/cron/window-reset.php
ℹ️ Find the correct PHP path via SSH: which php or which php8.1. Use the full path in your cron command.
2
Test the cron manually via SSH
Before relying on the scheduler, confirm the script runs without errors:
ssh user@yourdomain.com
/usr/bin/php /home/u123456/public_html/api/cron/window-reset.php
# Should output something like: "Window check complete. No action needed."
# Or: "New window created for 2026-04-17."

Stage 5 — Production QA Sign-Off

Run every item below against the live production URL. Do not skip any. Check off each item before announcing the form to customers.

Environment & Security

  • SSL active — padlock shows in browser on https://yourdomain.com
  • https://yourdomain.com/.env returns 403
  • https://yourdomain.com/api/helpers/db.php returns 403
  • APP_ENV=production in server .env (not sandbox)
  • Square SDK script tag points to web.squarecdn.com (not sandbox URL)
  • Square Production credentials confirmed in .env

Time Gate

  • Outside window: form hidden, countdown visible and ticking
  • Direct POST to /api/order.php outside window returns form_closed error
  • Inside window: form renders, Square card field loads
  • Status endpoint (/api/status.php) returns correct open boolean

Square Payments (Live)

  • $0.01 test charge completed successfully — order record created in DB
  • Order confirmation email received in inbox (not spam)
  • Owner notification email received at OWNER_EMAIL
  • Square Dashboard shows the $0.01 charge
  • $0.01 test charge refunded in Square Dashboard
  • Webhook payment.completed event shows 200 response in Square Dashboard

Cap Enforcement

  • Status endpoint shows correct rolls_remaining and orders_remaining
  • Admin dashboard cap counters reflect live DB state
  • Force Close toggle works — form closes within 60 seconds
  • Cap override fields save and take effect immediately

Admin Dashboard

  • Admin login works at /admin/login.php
  • Dashboard shows correct orders, revenue, and cap totals
  • Order search by name/email returns correct results
  • CSV export downloads a properly formatted file
  • Logout works — session destroyed, redirect to login

Mobile & Cross-Browser

  • Form tested on iOS Safari — Square card field renders and submits
  • Form tested on Android Chrome — same
  • Form tested on desktop Chrome and Firefox
  • Admin dashboard usable on mobile (basic responsive layout)

Rollback Plan

If a critical issue is discovered after launch, here's how to take the site back to a safe state without losing data.

Option A — Soft Close

Force Close via Admin Dashboard

Log into /admin/ → toggle Force Close. The form closes within 60 seconds for all visitors. No file changes needed. Use this for minor issues that need a quick fix.

Option B — Hard Redirect

Replace index.html with a holding page

Upload a static index.html with a "We'll be back soon" message via SFTP. Overwrite the live form. No server restarts needed on Hostinger shared hosting.

Option C — Full Rollback

Restore previous file set via SFTP

Keep a dated backup of your project folder locally. If you need to fully revert:

# In FileZilla — upload previous working version over the broken one
# Database is NOT touched — orders already placed are safe
# Only rollback DB if schema change caused the issue

# To export DB backup before any risky change:
# phpMyAdmin → Export → Quick → SQL → Go
